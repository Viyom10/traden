Title: How a Single Ed25519 Signature Becomes the Enforcement Mechanism

  steps, in EXACTLY the order shown below. Do NOT omit, merge, reorder,

• Do NOT replace the numbered list with floating "chips" or a horizontal
  flow — it must be one tall column on the LEFT half of the slide.

CENTRE-LEFT — vertical numbered pipeline (10 steps, in this order):

   1.  User clicks "Place order" on /perps
   2.  Drift SDK builds a VersionedTransaction (V0) for the trade
   3.  My DriftClientWrapper intercepts driftClient.sendTransaction
   4.  I resolve every Address Lookup Table referenced by the message
   5.  I decompile the V0 message into a TransactionMessage
   6.  I prepend SystemProgram.transfer(payer → builderAuthority, feeLamports)
   7.  I recompile the message with the SAME payer, blockhash, and ALTs
   8.  crypto.subtle.digest("SHA-256", message) → 32-byte digest
   9.  Phantom signs that digest with the user's Ed25519 secret key
   10. Solana validators verify the signature over the FULL message
       and execute every instruction atomically

   ─── outcome banner (below step 10, full-width) ───
   ✅ both succeed   |   ❌ both revert

RIGHT — three short "why this works" paragraphs:

  • Ed25519 signatures cover the entire message, byte-for-byte. Flip one
    bit anywhere — including the fee amount, the recipient pubkey, or
    the order of instructions — and SHA-256's avalanche property
    guarantees a different digest, which the validator's verifier
    rejects.

  • Solana's runtime executes the instruction list of a transaction
    inside a single account-state commit. There is no scheduling point
    between instructions where one can land without the other. This is
    a Solana property I am consuming, not inventing.

  • Combining the two: the fee is no longer a separate event at all.
    It is one of the bytes whose hash the user signed. Removing it
    changes the hash; modifying it changes the hash; reordering it
    changes the hash. Every adversary action invalidates the signature.

      a TransactionMessage, prepends a SystemProgram.transfer instruction,
      recompiles with the same payer / blockhash / ALTs, and forwards.

    tradingFee.ts
      Computes feeLamports = orderSize × 5/10000. For USDC-quoted markets,
      reads the live SOL/USDC price from Drift's oracle cache (Pyth +
      Switchboard) and converts the USDC fee to lamports before injection.

    cpi.ts
      Pure-TS extractor: turns a ParsedTransactionWithMeta's flat
      meta.innerInstructions list into a tree {parent → CPI children},
      with friendly labels for System / Drift / SPL Token / Pyth /
      Switchboard. Powers the call-graph view on /explorer.

    merkle.ts
      SHA-256 Merkle tree from scratch. Domain-separated leaf (0x00 ‖ x)
      and node (0x01 ‖ L ‖ R) hashing to defeat second-preimage attacks.
      Exposes buildMerkleTree, getProof, verifyProof. Backs Section 4 of
      /verify.

    solscan.ts
      Env-aware Solscan URL helpers (devnet vs mainnet). Used everywhere
      a signature is rendered so any reviewer can independently verify.
